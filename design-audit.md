## 새로 발견한 결함 (이번 점검에서 처음 확인)

### 🔴 `.capbar` 클래스명 충돌 — 홈 피드의 진행바가 실제로는 옐로우로 렌더링됨
Phase E에서 홈 피드(`.opencard`)용으로 `.capbar`/`.capbar i`를 새로 정의했는데(§145행대), 이미 참가 신청 페이지(`.poster` 히어로의 모집률 바)가 **같은 클래스명**을 쓰고 있었다(§273행대). CSS는 소스 순서상 나중에 나온 규칙이 이기므로, 신청 페이지 쪽 정의(`background:var(--sun)`)가 홈 피드 쪽까지 덮어써서 **홈 피드 카드의 모집 진행바가 파란색이 아니라 옐로우로 표시된다.** 실제 브라우저에서 계산된 스타일을 확인해 확정했다(`rgb(247,207,92)` = `--sun`).

```
167:.capbar{margin-top:14px;background:var(--main-soft);height:8px;border-radius:99px;overflow:hidden}   ← Phase E 홈 피드용 (의도: 파랑)
168:.capbar i{display:block;height:100%;background:var(--main);border-radius:99px}
...
273:.capbar{background:rgba(255,255,255,.22);border-radius:99px;height:14px;margin-top:16px;overflow:hidden}  ← 기존 신청 페이지용
274:.capbar i{display:block;height:100%;background:var(--sun);border-radius:99px;transition:width .5s}
```
→ 두 용도가 클래스명만 같고 시각적 의도(트랙 색, 두께)가 다르다. 신청 페이지 쪽을 `.capbar2` 등으로 분리하거나 홈 피드 쪽 이름을 바꿔야 한다. **옐로우 사용 규칙 위반이자 실제 시각 버그.**

### 🟡 `.brow .rank` — 전광판 순위 숫자가 모든 행에서 옐로우
```
300:.brow .rank{font-family:'Oswald',sans-serif;font-size:28px;font-weight:700;color:var(--sun)}
```
1위 뿐 아니라 2위·15위 등 모든 순위 숫자가 옐로우다. `.brow.top1`에만 배경 그라디언트가 별도로 걸려 있어(308행) 1위 강조 자체는 있지만, 숫자 색은 전원 옐로우라 "1등 전용" 규칙과 충돌한다.

### 🟡 `.bchip.on` — 전광판 필터 칩(코스 선택/남녀 탭) 선택 상태가 옐로우
```
311:.bchip.on{background:var(--sun);color:#5c4708;border-color:var(--sun)}
```
"종합/1코스/2코스" 탭이나 기간형 전광판의 "전체/남자/여자" 탭 등 **단순 선택 상태 표시**에 옐로우를 쓰고 있다. 1등·내 것과 무관한 사용.

### 🟡 `.poster .pdate` — 신청·요강 페이지 상단 상태 라벨이 옐로우
```
270:.poster .pdate{color:var(--sun);font-weight:600;font-size:16px;letter-spacing:.06em;margin-bottom:8px}
```
"접수중"/"진행중 LIVE"/"종료" 같은 상태 라벨 색상. 브랜드 포인트 색으로 쓴 것으로 보이나, 엄밀히는 1등·내 것 규칙 대상이 아니다.

### 🟡 `.holetop .hno span` / `.dots i.c` — 심판 입력 화면의 "PAR 3" 텍스트, 현재 홀 표시점
```
221:.holetop .hno span{font-size:16px;color:var(--sun);font-weight:600;letter-spacing:.06em}
234:.dots i.c{outline:3px solid var(--sun);outline-offset:1px}
```
파 정보 텍스트, 진행 중 홀 표시에 옐로우 — 위와 동일하게 강조 대상이 아닌 곳에 사용.

**옐로우 사용 중 규칙에 정확히 맞는 것**: `.minecard`/`.minecard .mk`(홈 "내 대회"), `.pod.first`(결과 1위 포디움), `.brow.top1` 배경(전광판 1위 행), `.avatar.lead`(위저드 대표운영자) — 이 4곳은 문서 규칙대로다.

### 🟡 폰트 굵기
`font-weight` 800/900을 인라인·클래스 어디서도 찾지 못했다(grep 결과 0건). **굵기 규칙은 현재 위반 없음.** 단, `Black Han Sans`(굵기 개념이 없는 디스플레이 폰트)가 전광판의 `.boardtop h3`, `.brow .name`에 쓰이는데, 이는 설계서에 이미 "전광판 전용 예외"로 명시된 항목이라 위반이 아니다.

## 화면별 상세

### ① 홈 — 부분 적용
- 반영됨: LIVE 다크 카드, 접수중 D-day 카드, 내 대회 옐로우 카드, 인사말, 필터 칩 유지.
- 남은 문제: 위 `.capbar` 충돌로 진행바 색이 잘못됨(위 참조). 시안은 LIVE 카드에 실시간 미니 리더보드(상위 3명)를 보여주지만, 현재 홈 로딩 로직은 참가자 수만 갖고 있어 미니 리더보드가 없다(의도적 보류 — 새 데이터 조회가 필요해 이번 재설계에서 제외했었음).

### ② 운영 허브 — 부분 적용
- 반영됨: `.stepper` 단계 스테퍼, `.stats` 숫자 카드 3개, 대회 요강 바로가기, "결과·시상 확정하기" 단일 버튼.
- 남은 문제: 시안의 `.tools`/`.trow`(아이콘+한 줄 요약+상태+화살표) 리스트 형태로 안 바꾸고, 심판 링크/전광판/신청 링크는 **기존 카드+개별 목록 UI를 그대로 유지**했다(§600~643행, `<div class="card">` 3개). 즉 운영 허브의 절반(스테퍼·통계)만 시안 스타일이고 나머지 절반(도구 카드들)은 예전 스타일이라 화면 하나에 두 가지 문법이 섞여 있다.

### ③ 대회 개설 위저드 — 미착수
- 시안: 한 화면 한 질문(`.wq`/`.optbig` 큰 선택지 카드), 상단 진행바(`.wprog`).
- 실제(§359~450행대, `#wp1`/`#wp2`/`#wp3`): 여전히 **한 화면에 대회명·주최·유형·기간·경기규모·코스·정원·동점처리·단계수·컷 설정까지 폼 필드가 전부 몰려있는 방식**. `.wizsteps`(1/2/3 라벨)는 있지만 시안이 말하는 "질문 하나씩 대화형으로"와는 구조가 다르다. `.optbig` 스타일 큰 선택 카드도 없음(필드 유형·단계수는 `.modew` 2~3버튼 토글로만 구현).

### ④ 전광판 — 기간형(예선 리더보드) — 부분 적용
- 반영됨(D2에서 구현): 남/여 세그먼트 탭, 코스 완주 현황(n/코스수), 컷라인 점선 표시.
- 남은 문제: 탭 선택 강조·순위 숫자에 옐로우를 써서(위 `.bchip.on`, `.brow .rank`) 옐로우 규칙 위반. 시안의 `.seg`(둥근 필박스 세그먼트, 흰 배경 강조) 대신 기존 `.bchip`(둥근 테두리 칩) 스타일을 그대로 재사용해 시안과 톤이 다르다.

### 전광판 — 라운드(필드) 모드 — 미착수
- 시안에 해당 화면 예시 없음(참고 대상 아님). 옐로우 규칙 위반(`.brow .rank`)은 기간형과 공유하는 CSS라 동일하게 적용됨.

### 참가자 등록 — 미착수
```
661:<table id="prevTable"><thead></thead><tbody></tbody></table>
671:<table id="ptable"><thead></thead><tbody></tbody></table>
```
명단 미리보기·참가자 목록 둘 다 `<table>`. "표 신규 사용 금지" 이전에 **기존 표조차 카드로 전환하지 않은 상태**. 화면 자체를 아직 손대지 않았다.

### 조편성 — 완료(문제 없음)
`.gcard` 카드 목록만 사용, `<table>`·kv행 없음. 시안에 해당 화면이 없어 옐로우/굵기 규칙만 확인했고 위반 없음(옐로우 미사용). 리디자인 대상으로 아직 지정되지 않았을 뿐, 규칙 위반은 없다.

### 결과·시상 — 부분 적용
- 포디움(`.pod`)·시상 카드는 이미 카드형이고 `.pod.first`에 옐로우를 정확히 1위에만 사용(규칙 준수).
- 전체 순위표는 여전히 `<table>`(§693행, `#rtable`):
```
693:<table id="rtable"><thead><tr><th class="tac">순위</th><th>이름</th><th>클럽</th><th class="tac">합계</th><th class="tac">타수차</th></tr></thead>
```

### 심판 입력 화면 — 미착수
홀 스코어 입력 UI 자체는 시안에 없는 화면. 옐로우 규칙 위반 2건(`.holetop .hno span`, `.dots i.c`, 위 참조) 외에 구조적 변경은 검토되지 않음.

### 참가 신청 페이지 — 미착수
`.poster` 히어로는 이미 카드형이라 kv행/표 문제는 없으나, `.pdate` 옐로우 사용(위 참조)과 `.capbar` 충돌의 원인 제공처.

### 대회 요강 페이지 — 미착수
참가 신청과 동일하게 `.poster`+`.card`+`.fl` 라벨 조합으로 이미 kv행/표는 없음(설계서 문서에도 "모범 사례"로 적었던 화면). `.pdate` 옐로우 사용만 공통 문제.

## 우선순위 제안 (참고용 — 이번엔 수정하지 않음)
1. `.capbar` 충돌 — 실제 버그이므로 최우선 수정 후보.
2. 옐로우 오용 5건(`.brow .rank`, `.bchip.on`, `.poster .pdate`, `.holetop .hno span`, `.dots i.c`) — 디자인 규칙 정합성 문제.
3. 위저드 대화형 전환(③) — 시안과 구조적으로 가장 크게 벌어진 화면.
4. 참가자 등록 `<table>` 2건, 결과 화면 `<table>` 1건 — 표 폐지 규칙 적용 대상.
