## ① 표 폐지 — 완료 (잔여 0건)

`<table` 전체 검색 결과, 실제 코드에 남은 표는 1건뿐이다:

```
933: <table id="rtablePrint" class="printonly">
```

이는 결과 화면의 **인쇄용 결과지 예외**(설계서에 명시된 유일한 허용 사례)다. 화면에는 `display:none`으로 숨겨져 있고 `@media print`에서만 노출되며, 실제 순위는 `#rankList`의 리스트 행으로 표시된다.

이번 재점검에서 새로 발견해 함께 정리한 것: 참가 신청(공개 링크) 페이지의 "단체 신청" 명단 인식 미리보기가 `#apPrevTable`이라는 표로 남아 있었다. v2.1~v2.2의 7개 차수 중 어디에도 이 화면이 명시적 범위에 없어 그동안 누락됐던 것이다. `.toolrow`(아바타+이름+전화번호) 리스트로 전환해 정리했다.

## ② 옐로우 사용 — 완료 (위반 0건)

`--sun`/`#F7CF5C` 전체 사용처를 확인한 결과, 전부 "1등" 또는 "내 것" 맥락에 한정된다:
`.avatar.lead`(위저드 대표운영자), `.livecard .lp.top1 .lrank`(홈 LIVE 카드 미니 리더보드 1위), `.minecard`/`.minecard .mk`(홈 "내 대회"), `.mycard`(내 대회 목록의 진행중 카드 강조), `.brow.top1 .rank`(전광판 1위), `.pod.first`(결과 포디움 1위), `.rrow.top1 .rk`(결과 순위 리스트 1위).

v2.1 Phase E에서 지적됐던 5건의 오용(`.brow .rank` 전체, `.bchip.on`, `.poster .pdate`, `.holetop .hno span`, `.dots i.c`)은 모두 이전 차수에서 수정되어 재발하지 않았다.

## ③ 폰트 굵기 — 완료 (위반 0건)

`font-weight:800` / `900` 검색 결과 0건. Black Han Sans(전광판 제목 전용, 굵기 개념 없는 디스플레이 폰트)는 기존 예외 그대로 유지.

## ④ 홈 = 피드형 + 3탭 내비 — 완료

- 하단 내비 3탭 확정: **홈**(`opHome`, 깃발) · **내 대회**(`opMine`, 트로피) · **MY**(`opMy`, 사람). 예전 5탭(대회/운영/참가자/조편성/결과) 구조 잔재 없음.
- 홈은 인사말 + 필터칩 + `mineSection`(내 대회 카드) + `listSection`(LIVE 다크카드·접수중 카드) + 하단 "새 대회 개설하기" 버튼 — 피드 구조 그대로.
- 내 대회 탭: 0개(빈 화면+개설 유도) / 1개(목록 생략, 바로 진입) / 2개 이상(카드 목록) 분기 정상.

## ⑤ 위저드 = 대화형 + 신규 질문 — 완료

`wizardSteps()`가 반환하는 순서(1단계 기준):
`wpBasic → wpType → wpEligibility → wpVisibility → wpStageCount → wpStage1 → wpCap → wpParticipantInfo → wpAwards → wpNoticeExtra → wpNoticePreview → wpStaff`
(2·3단계는 `wpStage2`/`wpStage3`이 `wpStage1` 뒤에 추가로 끼어든다.)

배열에 있는 모든 pane id가 실제 `.wpane` div로 존재하고, 반대로 배열에 없는 고아 `.wpane`도 없다. 참여 대상(`wpEligibility`)·공개 설정(`wpVisibility`, 비밀번호 형식 검증 포함)·시상 개편(총상금+주요 시상 추가식 목록)·요강 부가정보(`wpNoticeExtra`, 전부 선택)까지 10문항 전부 한 화면 한 질문 구조로 반영되어 있다.

## ⑥ 관리공간 7탭 — 완료

`.mtabs`의 7개 탭: **관리홈·요강·참가자·운영진·스코어입력·리더보드·결과**(`mgHome`/`mgNotice`/`mgPlayers`/`mgStaff`/`mgScore`/`mgBoard`/`mgResult`). 전부 내용이 채워져 있고, 대회명·요약이 상단에 고정되어 어느 탭에서도 컨텍스트가 유지된다. 조편성은 7탭에는 없지만 참가자 탭의 세그먼트로 정상 흡수됨(§7 참고).

## ⑦ 참가자 3분할 + 스코어입력 — 완료 (의도적 축소 2건 남음)

참가자 탭 세그먼트 `명단(pList) · 조편성(pGroups) · 대기(pWait)` 모두 정상 동작. 스코어입력 탭도 라운드형(조별 진행 현황, 완료=민트/미시작=코랄, 조 탭→홀별 입력)과 기간형(선수 검색→코스→총타수) 분기가 확인된다.

다만 개발 과정에서 의도적으로 축소한 2가지는 여전히 남아있다(회귀나 누락이 아니라, 백엔드 지원이 없어 해당 차수 진행 시 명시적으로 보고했던 항목):
- "스코어카드 사진 입력" 행에 "검수 대기 N건" 뱃지가 없음 — 승인/반려 상태 컬럼이 DB에 없어 구현하지 않음.
- 스코어입력 탭 자체는 있지만 조 진행 현황과 심판 화면이 완전히 하나로 통합되지는 않음(조 탭 시 기존 심판 화면으로 진입하는 방식).

두 항목 모두 신규 스키마·워크플로 설계가 필요해 이번 v2.2 범위를 넘어서므로, 다음 라운드의 별도 과제로 남겨두는 것을 제안한다.

## ⑧ 비공개 대회 처리 — 완료

- `loadTournaments()`가 `.or(visibility.eq.public, owner_id.eq.내id)`로 조회 — 비공개 대회는 서버 쿼리 단계에서 익명 방문자에게 아예 전달되지 않는다(클라이언트 은폐가 아님).
- `get_tournament_gated` RPC(security definer)가 요강·전광판·신청 페이지 직접 접속을 감싸 비밀번호가 맞아야 실제 데이터가 내려간다.
- 통과 시 기기(localStorage)에 코드가 저장되어 재입력 불필요. 심판 토큰 링크(`?j=`)는 게이트 없이 기존 경로 그대로 통과.
- 공유 문구(요강 링크 복사·전광판 주소 복사·신청 링크 복사)가 비공개 대회면 "🔒 입장 비밀번호: XXXX"를 자동 포함.

## ⑨ 배지 로고 — 완료

`brand/logo-badge.svg`가 로그인 화면 3단계, 신청 페이지 포스터, 요강 페이지 포스터, 접수완료 화면까지 총 6곳에 반영되어 있다. 앱바·전광판 푸터는 지시대로 심벌 단독(`logo-symbol.svg`) 유지 — 점검 중 이 두 곳이 삭제된 `symbol-white.svg`를 참조해 깨져 있던 것을 발견해 함께 고쳤다. `symbol-white.svg` 참조는 현재 0건.

## ⑩ 홈 LIVE 카드 top3 — 완료

`top3_standings` RPC가 서버에서 상위 3명만 계산해 반환하고, `loadTournaments()`는 `status==='live'`인 대회에 한해서만 이 가벼운 조회를 호출한다. 실데이터로 순위·타수가 정확히 표시되고 1위만 옐로우로 강조되는 것을 확인했다.

## 결론

v2.2 차수 0~7의 확인 조건을 모두 충족했고, 재점검 중 발견한 잔여 표 1건(`#apPrevTable`)도 함께 정리해 "표 폐지" 규칙의 잔여 위반이 0건이 되었다. §7에 적어둔 2가지 의도적 축소(검수 대기 뱃지, 스코어입력-심판화면 완전 통합)는 새 스키마/워크플로가 필요한 별도 과제로 남겨두는 것을 제안한다.

이제 워크오더에 명시된 대로 **출시 전 점검(성능·개인정보 동의·오프라인 대비) 단계**로 넘어갈 수 있다.
